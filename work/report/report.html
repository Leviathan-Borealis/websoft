A short description of browser - HTTP - web server interaction:
A user request a web page via a browser. The browser connects to the web server via a HTTP GET. The web server looks for the resource specified in the GET from the browser. If found the web server responds to the browser with the requested data and the return code 200. Upon receiving the data (string of html) from web server the browser searches the received html for external resources referenced. The browser performs request for these resources and when all resources are locally(there are exceptions to this of course) the page will be showed fro the user

S06

Native languages: These languages will always retain their niche due to performance and platform interaction capabilities Ex. Delphi, C/C++, Go etc.
Compiled languages: Compiled languages generally compile to C and then down to assembler/machine. These languages often retain some hardware interactions but the implementation is out of the developers control. These languages is nice to use if the middle and low-level implementations is beyond the scope of your project but you still want the platform compatibility. Ex. Common Lisp, GHC Haskell, Perl
Interpreted/Scripted: These languages are light and easier to approach than the native ones. The are generally platform independent which is a sell point in any business. The are generally abstracted from hardware and must utilize native language middleware to interact with hardware efficiently Ex. Java, Python, JavaScript

During this section I have been taking mental notes of how to avoid bottlenecking in my code. To enable vital code to load first and then incrementally add on functionality as the data becomes available. This produces a fast site that will have functionality with only partially loaded page.

S09

Implement https. Today this is almost a necessity but it's worth to be reminded. Using secure transfer makes your data private during transfer. It is still good practice to not push user-credentials in "clear-text" even on a secure connection.

Libraries is something most developers use frequently. Most of the time the are linked via Maven. But it is not common to review used libraries for security flaws och back-doors. I recently read an article about security issues within common libraries. Some flaws were just bugs. But some were deliberately inserted. To review included libraries is not something commonly performed in industry and this is a huge problem. The next best thing if not reviewing a library is to use well-known libraries/frameworks. Due to many users errors within these libraries/frameworks should be spotted earlier and removed. But there is never a guaranty

Speaking of libraries and frameworks. Until you know how to implement proper security by yourself it is best practice to use libraries and framework for this task. You will probably end up with better security than implementing by hand. Remember, an attacker need only one hole. You need to find all the holes...

Last but not least, never ever ever use a development environment for production. A dev environment is configured to make it easy to implement your site/software. But this also makes it easy for the hacker. Some popular servers have had to implement some default security in their development environment due to the fact that many put the server online in development configuration with disastrous consequences

I have been hosting 3 servers for some years and most of the server-side problems discussed was not new to me. But the part about sessions and tokens is interesting. I haven't implemented this kind of security on any website/app yet. We will probably do it in our project.
